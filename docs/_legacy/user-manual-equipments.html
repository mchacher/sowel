Equipments

Equipments are the user-facing building blocks of your smart home. They abstract away hardware details and give you meaningful, room-level controls like "Living Room Spots" or "Garage Door".

What is an Equipment?

While Devices represent physical hardware discovered from integrations (Zigbee2MQTT, Panasonic CC, MCZ Maestro…), Equipments represent what you actually interact with. An Equipment is a named, functional unit that binds to one or more Device data points.

📚
Think of it this way: A Zigbee motion sensor paired via Zigbee2MQTT is a Device. "Hall Motion Detector" displayed on your dashboard with its zone, icon and automations is an Equipment. The link between them is a Binding.

This separation is a core differentiator of Sowel: the user thinks in terms of "Garage Door" or "Kitchen Spots", not in terms of Zigbee addresses, MQTT topics, or protocol-specific values. Integrations handle the "dirty" mapping; equipments stay clean and universal.

Equipment Types

Each Equipment has a type that determines its icon, available controls, how it aggregates in zones, and how it appears on the dashboard.

Type Description Data (read) Orders (write)
light_onoff Simple on/off light state state
light_dimmable Light with brightness state, brightness state, brightness
light_color Light with color temperature state, brightness, color_temp state, brightness, color_temp
shutter Roller shutters, blinds position (0–100%) state (OPEN/CLOSE/STOP), position
gate Garage door, motorized gate state (open/closed/unknown) command
thermostat Heat pump, pellet stove, AC power, mode, setpoint, temperature power, mode, setpoint
sensor Temperature, humidity, motion, contact… temperature, humidity, motion, battery… —
button Remote, push button action (single/double/hold) —
weather Weather station (outdoor) temperature, humidity, wind, rain… —

✅
Tip: Choose the right type during creation — it defines the UI controls, zone aggregation behavior, and which device data categories are offered for binding. For example, light_dimmable exposes a brightness slider, while light_onoff only offers a toggle.

Creating an Equipment

  1. Navigate to Equipments
    From the sidebar, click Equipments. You'll see the list of all existing equipments, or a prompt to create your first one.
  2. Click "Add Equipment"
    Enter a name (e.g. "Spots Salon"), select the type (light_dimmable, shutter, sensor…), and assign it to a zone.
  3. Select Devices
    Pick one or more Devices from the list. The list is filtered to show only devices that match your equipment type. For example, selecting type light_dimmable will only show devices that expose light_state and light_brightness data.
  4. Auto-Binding
    Sowel automatically creates data bindings (read) and order bindings (write) based on the device capabilities. Each binding gets an alias — a stable name used everywhere in Sowel (automations, UI, API).
  5. Done
    Your equipment is now live. It appears on the dashboard, participates in zone aggregations, and can be targeted by scenarios and recipes.

Understanding Bindings

Bindings are the links between an Equipment and the raw Device data. Sowel uses two kinds:

  • Data Bindings (read) — the equipment reads a value from the device. Example: a Zigbee temperature sensor reports 21.5, and the "Office Sensor" equipment sees it as its temperature binding.
  • Order Bindings (write) — the equipment sends a command to the device. Example: you toggle "Kitchen Spots" ON, and the order binding dispatches the command to the Zigbee bulb via Zigbee2MQTT.

Data Binding Properties

Property Type Description
alias string Canonical name to reference this binding (e.g. state, temperature, brightness)
deviceDataId UUID The specific device data point to read from
category string Data category that drives zone aggregation and icon display (e.g. light_state, temperature, motion)
type string Value type: boolean, number, enum, text

Order Binding Properties

Property Type Description
alias string Canonical name to reference this order (e.g. state, brightness, command)
deviceOrderId UUID The specific device order to dispatch to
type string Value type: boolean, number, enum
enumValues string[] Available values for enum orders (e.g. ["OPEN","CLOSE","STOP"])
min / max number Value range for numeric orders (e.g. brightness 0–254)
✅
Tip: Aliases are stable identifiers. Recipes, scenarios, and the API all reference bindings by alias. When you create an equipment, Sowel auto-assigns meaningful aliases from the device data keys. You can rename them if needed.

Example: Dimmable Light

A Zigbee2MQTT dimmable bulb (e.g. IKEA TRADFRI LED1545G12) auto-bound as a light_dimmable equipment:

// Equipment: "Spots Salon" (type: light_dimmable) Data Bindings (read): "state" ← device data "state" // category: light_state, type: boolean "brightness" ← device data "brightness" // category: light_brightness, type: number Order Bindings (write): "state" → device order "state" // type: boolean (ON/OFF) "brightness" → device order "brightness" // type: number, min: 0, max: 254

Example: Gate

A Zigbee contact sensor on a garage door, auto-bound as a gate equipment:

// Equipment: "Porte Garage" (type: gate) Data Bindings (read): "state" ← computed // category: gate_state, values: "open" | "closed" | "unknown" "contact" ← device data "contact" // category: contact_door, type: boolean (raw) Order Bindings (write): "command" → device order // type: enum, enumValues: ["open", "close"] // The "state" binding is derived by the backend from the raw // contact sensor data. The UI only sees "open", "closed", or "unknown".
📚
Abstraction in action: The gate's state is computed by the backend from raw device data (a Zigbee contact sensor, relay reed switches, etc.). The UI never deals with hardware-specific details — it just reads state = "open" or state = "closed".

Value Resolution

When you execute an order on an equipment (toggle a light, send a gate command), Sowel resolves the value automatically when possible:

  • Single enum value — auto-resolved. A gate with enumValues: ["latch"] doesn't require you to specify the value; Sowel sends it automatically.
  • Multiple enum values — you choose. A shutter with ["OPEN", "CLOSE", "STOP"] shows buttons for each option.
  • Numeric range — slider or input. A brightness order shows a slider from min to max.
  • Boolean — simple toggle.

Zone Aggregation

When equipments are assigned to a zone, their data bindings contribute to zone-level aggregated values. The aggregation method depends on the data category:

Category Aggregation Zone Field Example
temperature AVG temperature "Living Room: 21.3°C"
humidity AVG humidity "Bathroom: 65%"
luminosity AVG luminosity "Office: 320 lux"
light_state COUNT on/total lightsOn, lightsTotal "3/5 lights on"
motion OR + COUNT motion, motionSensors "Motion detected (2 sensors)"
shutter_position AVG + COUNT averageShutterPosition "Shutters: 75% avg"
contact_door COUNT open openDoors "1 door open"
⚠
Note: Weather equipments are excluded from zone aggregation to prevent mixing outdoor readings with indoor data. Gates are not aggregated either (each is individual).

Button Actions

button equipments are special: instead of order bindings, they have action bindings that map press types to effects.

Press Type Available Effects
single Activate mode, toggle between two modes, send equipment order, toggle recipe
double Same options
hold Same options

For example, a Zigbee2MQTT IKEA remote (TRADFRI on/off switch) can be configured so that a single press toggles the living room lights, a double press activates "Movie" mode, and a long press turns off all lights in the zone.

Zone Orders (Group Commands)

Zone orders execute a command on all matching equipments within a zone at once:

Command Targets Description
allLightsOn light_onoff, light_dimmable, light_color Turn on all lights in the zone
allLightsOff light_onoff, light_dimmable, light_color Turn off all lights in the zone
allShuttersOpen shutter Open all shutters (position = 100%)
allShuttersClose shutter Close all shutters (position = 0%)
